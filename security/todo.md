## vault-config: bring forge-reader Vault policy under Crossplane management

**Status:** open. Currently `forge-reader` (used by ESO across all clusters)
is provisioned by `vault/playbooks/configure.yml` as part of Vault
bootstrap. With provider-vault in place, it should move into
`security/secrets/vault-config/values.yaml` as another entry under
`policies:` so its definition is git-push.

Migration path: add to vault-config values.yaml; on first reconcile,
provider-vault will detect the existing policy and 409 (similar to the
realm/clients adoption issue — see Q15 in identity/keycloak/QA.md).
Need `crossplane.io/external-name: forge-reader` annotation, OR delete
the existing policy in Vault first then let Crossplane create fresh.

## vault-config: bring oauth2-proxy/cookie-secret under Crossplane

**Status:** open. Currently seeded manually (`vault kv put kv/forge/oauth2-proxy/cookie-secret cookie_secret=...`) — same as the
crossplane-admin token in `kv/forge/crossplane/vault-token`. Both are
the only manual `vault kv put` operations left in the bootstrap.

Migration: provider-vault's `KVSecretV2` resource lets us declare KV
secrets as CRs. The cookie value would need to be generated by Crossplane
(or supplied as a literal). For homelab, a literal in
vault-config/values.yaml is simplest — then rotation = git commit + push.

Important: rotating the cookie invalidates all live oauth2-proxy
sessions. Should be infrequent.

## vault-config: also manage the kubernetes-mgmt-* auth backends

**Status:** future. Phase 8 of cluster bootstrap creates these mounts
imperatively via ansible (cluster/k8s-bstrp/tasks/register-vault.yml).
Long-term they could be Crossplane CRs (`AuthBackend.kubernetes.vault.upbound.io`)
declared in vault-config, with `token_reviewer_jwt` sourced from a Secret
on the target cluster (cross-cluster reads via provider-kubernetes
pointed at each remote cluster).

Cost: complex multi-provider plumbing. Benefit: Phase 8 ansible task
can shrink to "create SA only", everything else is git-push. Worth it
when more clusters come online.

## Trivy DB mirror in Harbor

**Status:** not started. Blocks reliable cluster bootstrapping at scale.

Trivy fetches its CVE DB from `ghcr.io/aquasecurity/trivy-db` on every cluster install. GitHub rate-limits unauth requests at ~60/hr, which is fine for one cluster but dies on bulk rebuilds. Set up Harbor as a pull-through cache for `ghcr.io/aquasecurity/*`. Done = trivy-operator on a fresh cluster reads from `harbor.apps.mgmt-forge.engatwork.com/aquasecurity/trivy-db` instead of GitHub.
